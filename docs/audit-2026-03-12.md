# Adversarial Audit — 2026-03-12

## Executive Summary

Normalize has grown from "three composable primitives" to a 38-crate ecosystem with
22 top-level CLI commands and 90+ entry points. The underlying architecture is sound
(no circular deps, clean trait boundaries, proper layering), but the documentation,
CLI surface, and main crate size haven't kept pace with the scope expansion.

**Structural health: Good.** No circular deps, clean DAG, minimal foundation crates.
**Documentation coherence: Poor.** Philosophy.md describes a different project.
**CLI coherence: Poor.** Overlapping commands, inconsistent naming, `analyze` is a dumping ground.
**Crate granularity: Good.** Right-sized crates with clear domains (a few exceptions).

---

## 1. Architecture: What's Working

- **No circular dependencies** across 38 crates — clean DAG
- **`normalize-languages`** (20k LOC, 9 dependents) is correctly the most-depended crate
- **Trait separation**: `Language` (syntax/AST) vs `LocalDeps` (filesystem/packages) — no cross-dep
- **Plugin architecture** for rules: syntax (tree-sitter queries), fact (Datalog), native, SARIF — unified through `normalize-rules`
- **Foundation crates are minimal**: `normalize-core` (234 LOC), `normalize-facts-core` (227 LOC)
- **Output formatting** (`OutputFormatter` + server-less) gives consistent `--json`/`--pretty`/`--compact` everywhere

## 2. The Main Crate Problem

**`normalize` is 67k LOC** — 25% of the total workspace.

| Slice | LOC | % of main |
|-------|-----|-----------|
| `commands/analyze/` | ~17k | 25% |
| `commands/view/` + other | ~8k | 12% |
| `service/` | ~12k | 18% |
| rest (config, serve, etc.) | ~30k | 45% |

CLAUDE.md says "main crate is for CLI wiring" but 46 analysis commands with report structs
and formatters is beyond wiring. `normalize-analyze` exists (326 LOC) but only holds shared
types — the actual command logic hasn't moved.

**Action**: Extract algorithms from `commands/analyze/` into domain crates individually.
Don't bulk-move; extract when a second consumer exists or when the algorithm is genuinely
reusable. The service layer stays in main.

## 3. CLI Surface: Incoherent

### 3.1 `analyze` is a dumping ground (40+ subcommands)

Six distinct domains forced under one namespace:

| Domain | Commands |
|--------|----------|
| Code quality | complexity, length, duplicates, fragments, uniqueness, ceremony |
| Module structure | size, density, module-health, imports, depth-map, surface, layering |
| Git history | coupling, hotspots, ownership, contributors, activity |
| Graph analysis | call-graph, trace, architecture, graph, dependents |
| Testing | test-ratio, test-gaps, budget |
| Tree-sitter introspection | parse, query, node-types |

A new user looking for "module layering" won't think to check `analyze layering`.

### 3.2 Duplicate commands

| Command A | Command B | Notes |
|-----------|-----------|-------|
| `syntax ast` | `analyze parse` | Both show tree-sitter CST |
| `syntax query` | `analyze query` | Both run tree-sitter queries |
| `analyze check` | `rules run --engine native` | `analyze check` should be deleted |

### 3.3 `facts` vs `structure` naming confusion

- CLI command: `normalize facts` (current)
- MEMORY.md: "`structure` replaces `facts`"
- Crate names: all `normalize-facts-*`
- Docs: both terms used interchangeably

Pick one. Commit everywhere.

### 3.4 `view` has 29 flags

This command does: file viewing, symbol navigation, dependency resolution, git history,
import inlining, skeleton extraction. Flag groups should be documented as mutually
exclusive, or the command should decompose.

### 3.5 Inconsistent patterns

- Domain-first (good): `edit delete`, `daemon start`, `grammars list`
- Verb-first (inconsistent): `generate client`, `generate types`
- Single-use top-level: `aliases`, `context`, `update` — should be subcommands or flags

## 4. Documentation: Tells the Wrong Story

### 4.1 Philosophy.md vs reality

| Philosophy claim | Reality |
|-----------------|---------|
| "Three primitives, fits in working memory" | 22 commands, 90 entry points |
| "Index is optional, builds on demand" | Hard-fails without index for cross-file features |
| "Hyper-modular" | 67k LOC main crate |

Philosophy.md isn't wrong architecturally — the *principles* are sound. But the scope
grew far beyond what the document describes, and it was never updated to acknowledge
the expanded surface.

### 4.2 Stale references

- cli-design.md references `syntax rules` (now just `rules`)
- README claims full ecosystem support; TODO.md shows bulk indexing is unresolved research
- TODO.md items marked ~~DONE~~ that are actually "scaffolded but not wired"

### 4.3 TODO.md is a 1047-line brain dump

Mixes 1-hour fixes with years-of-work vision. No prioritization, no dates, no
acceptance criteria. "When Ready / First Release" section is a 3-line stub.

### 4.4 MEMORY.md exceeds its own limit

333 lines vs 200-line cap. Second half truncated and unreachable. Design decisions
stored there (subagent worktree patterns, scope analysis details) are lost.

## 5. Crate Overlap Assessment

**No real overlaps.** Apparent similar names have clear separation:

- `normalize-deps` (module deps from code) vs `normalize-local-deps` (package discovery from filesystem) vs `normalize-ecosystems` (ecosystem management) — different layers
- 5 rule crates — different execution engines, appropriately separated
- `normalize-manifest` → `normalize-local-deps` → `normalize-ecosystems` → `normalize-package-index` — clear hierarchy

**One concern**: `normalize-analyze` (326 LOC type crate) exists alongside 17k LOC of
actual analysis commands in main. Either use it or delete it.

---

## Verdict

The architecture is sound. The implementation is solid. The *surface* — docs, CLI naming,
command organization — hasn't kept up with the scope expansion. The underlying code doesn't
need restructuring; the user-facing layer needs rationalization.

The highest-leverage fix is updating philosophy.md to match reality. Everything downstream
(CLI design, naming, docs) follows from having an accurate vision statement.
