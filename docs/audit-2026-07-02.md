# Main-Crate Decomposition Audit — 2026-07-02

Follow-through on the `docs/audit-2026-03-12.md` open P2 item *"Extract analysis
algorithms from `commands/analyze/` into domain crates (individually, not bulk)."*

## Executive Summary

**The main `normalize` crate is NOT a reservoir of extractable domain logic.** The
reusable algorithms — MinHash/LSH clone detection, cyclomatic complexity, symbol
extraction, model pricing, refactor transforms, the liveness solver — are **already**
in feature crates. What remains in `commands/` and `service/` is overwhelmingly
legitimate compute-and-format wiring that correctly stays in main under the extraction
bar (single consumer, no standalone value). The ~84k-line size of the main crate is
**CLI-surface breadth, not leaked domain logic.**

This partially **refutes** the handoff premise that "the main crate holds domain logic
that belongs in crates." It does not — the March P2 item is effectively already
satisfied. What the audit *did* surface is a short list of small duplications and one
encapsulation leak, plus two judgment-call extractions and a rename. None is a large
structural move.

**Structural health: Good.** Domain logic lives in crates; main holds wiring.
**Duplication: Minor.** Six narrowly-scoped dedups/extractions, mostly S-effort.
**Naming: One smell.** `normalize-analyze` is misnamed (it is the rank/render layer).

## Correction (2026-07-02): the vendored CLIs measure line-count, not cost

This audit — and the session analysis around it — leaned on the ~84k-line size and, in
framing what accounts for it, treated the vendored full-CLI front-ends (`src/rg/` ~13.9k,
`src/ast_grep/` ~7.0k, `src/jq/` ~0.7k; ~21.6k lines, ~26% of the main crate) as if they
were a size-reduction *lever*. That framing conflated **line count with cost**, and the
correction is worth recording precisely.

The vendored CLIs are **not a cost-reclamation target.** The real cost is the engine
libraries (`jaq-*`, `grep`/`ignore`, `ast-grep-*`), and those are **sunk cost** — already
linked into the binary because normalize's first-class features (`--jq`, `normalize grep`,
ast-grep via `normalize_syntax_rules`) require them, independent of whether the vendored
front-ends exist. Given the engines are already paid for, the marginal cost of also exposing
their full CLI front-ends is near-zero: the ~21.6k lines are mostly declarative flag tables
on top of already-linked engines. Extracting them would **not** meaningfully shrink binary
or compile cost. (The full marginal-cost rationale is now documented in
`docs/cli-dropin-integrations.md` under "Why the line count is not the cost.")

What this correction does **not** do — it does not dismiss the size question. The
line-count / encapsulation / SRP concern is real (~21.6k lines of verbatim third-party
tool front-ends living in the main crate is a single-responsibility smell, independent of
the negligible marginal binary cost). The remaining question was whether that smell
warrants *extraction into separate crates*. That question is now **decided** — see below.

## Decision (2026-07-02): vendored-CLI extraction is REJECTED — keep in main

The previously-open "extract the vendored CLIs (`src/rg/`, `src/ast_grep/`, `src/jq/`,
~21.6k lines) into their own crates" follow-up — earlier gated on (1) version lockstep and
(2) whether verbatim vendored wrappers can be published — is **closed as a negative
decision: do not extract.** The vendored CLIs stay in the `normalize` crate. The two gating
conditions are therefore **moot** (there is no extraction to gate). Reasoning, recorded so
this is not re-litigated:

1. **The only argument for extraction was SRP / encapsulation — there is no cost payoff.**
   As established in the correction above, the engine libraries are sunk cost for the
   first-class `--jq` / `grep` / ast-grep features; extraction reclaims no binary or compile
   cost. So the entire case for a crate rested on encapsulation aesthetics alone.
2. **A crate fails the crate-existence bar in every form.** CLAUDE.md requires a crate to
   have **(a)** multiple *actual* workspace dependents **or** **(b)** genuine standalone
   value. A vendored copy of ripgrep/jaq/ast-grep would have exactly **one** dependent
   (`normalize`) and **zero** standalone value — it is a verbatim copy of an already-published
   upstream tool, so nobody would depend on it instead of ripgrep/jaq/ast-grep themselves.
   `publish = false` does **not** rescue this: a private crate still must earn its existence
   by the same bar. The proposed remedy fails the very rule that governs crates.
3. **Naming is incoherent.** `normalize-rg` / `normalize-jq` / `normalize-ast-grep` would
   misrepresent verbatim third-party tool copies as normalize-domain components; you cannot
   republish ripgrep under a renamed crate, and no coherent name exists — a symptom that
   there is no coherent crate to extract.
4. **The SRP smell is already adequately mitigated without a crate.** The vendored code
   lives in isolated module subtrees (`src/rg/`, `src/ast_grep/`, `src/jq/`) and is gated
   behind clean capability feature flags (`cli-full` / `jq-cli` / `rg-cli` / `ast-grep-cli`).
   A crate would add an incoherent name and a bar-failing artifact for zero real gain.

Net: the vendored CLIs stay in `normalize`. The marginal-cost rationale (above and in
`docs/cli-dropin-integrations.md`) explains why they are cheap to keep; this decision closes
the "should we extract them" question with **no, and here's why.**

## Methodology

Five focused cluster audits were run against the strict CLAUDE.md extraction bar —
extract **only** if:

- **(a)** multiple *actual* workspace dependents exist, **or**
- **(b)** the code has *genuine standalone value* (could be published and used without
  normalize).

"Could theoretically be reused someday" does **not** clear the bar.

Clusters audited: (1) similarity/clone detection, (2) sessions analytics, (3) CFG
dataflow, (4) scoring/extraction, (5) service layer. Each candidate was checked with
`grep` for actual dependents and for a pre-existing home before proposing a move.

### 1. Liveness dataflow — EXTRACT → `normalize-facts`

Confirmed near-verbatim duplicate. The backward-dataflow fixed-point
(`live_in = use ∪ (live_out − def)`, reverse iteration) at
`commands/analyze/liveness.rs:246-294` is copied at
`normalize-refactor/src/extract_function.rs:700` (`compute_liveness`); both re-query the
same four cfg tables (`cfg_blocks` / `cfg_edges` / `cfg_defs` / `cfg_uses`) with the
same loader shape. **2 real workspace dependents → clears bar (a).**

Moves: the pure solver (~50 LOC) plus the four cfg-table loaders (~130 LOC).
Stays in main: `BlockLiveness` / `LivenessReport` + `OutputFormatter`.

Target is `normalize-facts` (owns the tables + libsql), **not** `normalize-cfg` (an
in-memory `Cfg` model that would be mis-shaped by a DB-backed pass). **Effort M / risk S.**

### 2. LSH candidate-pair loop — EXTRACT → `normalize-code-similarity`

The "bucket signatures per band, emit within-bucket pairs, merge+dedup" loop over
`&[[u64; MINHASH_N]]` → `Vec<(usize, usize)>` is copy-pasted 3× (`duplicates.rs`
~937-985 and ~1648-1668; `fragments.rs` ~587-620). **3 in-workspace dependents.**
Belongs beside the existing `lsh_band_hash` in the crate. ~25 LOC. **Effort S / risk S.**

### 3. Hand-rolled `UnionFind` — DEDUP

`clusters.rs:48-79` and `duplicates.rs:577-604` each re-implement find/union despite
`normalize_code_similarity::UnionFind` already existing. Collapse both onto the crate's.
**Effort S / risk trivial.**

### 4. `aggregate_sessions` fold — RESHAPE → `normalize-session-analysis`

`commands/sessions/analyze.rs` `aggregate_sessions` (~170 LOC) reaches into
`SessionAnalysisReport` internals (`message_counts` / `tool_stats` / `token_stats` /
`tool_chains` / `largest_tool_results`) from main to fold N reports into one — an
encapsulation leak. Move the ~80-120 LOC fold into session-analysis as
`aggregate_reports` / `SessionAnalysisReport::aggregate`; the print/dispatch wrapper
stays in main. **Effort S / risk S.**

### 5. heatmap `normalize_path` — DEDUP

`commands/sessions/heatmap.rs` holds a byte-for-byte copy of the already-`pub`
`normalize_session_analysis::normalize_path`. Call the crate's. **Effort S / risk trivial.**

### 6. Extraction-fixture harness — EXTRACT (judgment call) → `normalize-facts`

~380 LOC golden-diff engine embedded in `service/facts.rs:1209-1590`
(`run_extraction_fixture_case`, `discover_extraction_fixture_cases`, `find_input_file`,
`ExtractionExpected` / `ExpectedSymbol` / `ExpectedImport` / `ExpectedCall`). Parses via
`normalize_facts::SymbolParser`, diffs against `expected.json`. Homeless (grep confirms
no crate exposes a fixture harness); clears the standalone-value bar (b) but has only
1 caller, so **not** bar (a). The judgment-call extraction is justified by the coherent
380-LOC module weight. The `test_fixtures` `#[cli]` method stays as a thin wrapper.
**Effort M / risk S.**

### Rename: `normalize-analyze` → `normalize-rank`

The crate is **not** dead — the March "use it or delete it" flag is resolved. It is
consumed via `impl normalize_analyze::Entity for FunctionComplexity` plus `RankEntry` /
`format_ranked_table`. Its only smell is the name: it holds entity types + the rank
pipeline + table rendering, not analysis. Rename to reflect that it is the shared
rank/render layer. **Effort S / risk S.**

## Explicit Non-Extractions

Recorded so they are not re-litigated as negative decisions:

- **Keep BOTH session crates.** `normalize-chat-sessions` (parsing) vs
  `normalize-session-analysis` (per-session analytics) is a coherent, documented split,
  not redundancy. Do not consolidate.
- **`normalize-cfg` stays as-is** — a pure in-memory `Cfg` model; the wrong home for a
  DB-backed dataflow pass (see finding 1).
- **STAYS (legit compute+format for one command, no standalone value):**
  - `health.rs` — scoring is domain *policy* but scores the report's own fields for one
    command; the complexity input is already in `normalize-facts`.
  - `skeleton.rs` — thin wrapper over `normalize_facts::Extractor` + a `ViewNode` adapter.
  - `extract.rs` — already re-exports `normalize-facts`.
  - top-level `analyze/` — calls `normalize_facts::extract::compute_complexity` (not
    re-implemented); presentation/taxonomy only. `normalize-metrics` is a lean `Metric`
    trait, **not** a complexity engine — do not fold in.
  - `commands/analyze/` suppression heuristics, all `build_*_report` fns, and
    `duplicates_views.rs` (presentation); `fragments.rs` tree-walk collectors;
    `exceptions.rs` / `provenance.rs` / `trace.rs` / `call_complexity.rs` (one-command
    surfaces).
  - `service/edit.rs` — correctly delegates every transform to
    `normalize-refactor` / `normalize-edit` (verbose but genuine wiring).
  - `service/facts.rs` command core — delegates to `normalize-facts`.
