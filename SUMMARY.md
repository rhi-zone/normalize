# Normalize Monorepo

Normalize is a polyglot code intelligence CLI toolchain providing structural analysis of codebases through AST-based extraction. The monorepo contains 38 published Rust crates plus two `publish = false` crates (`normalize-grammars`, `xtask`), a VS Code extension (`editors/`), a web sessions viewer (`web/`), a documentation site (`docs/`), and supporting scripts and tooling. The main `normalize` binary consumes a library ecosystem of domain crates covering language support, facts extraction, rules evaluation, manifest parsing, and output formatting. Core development conventions are in `CLAUDE.md`; architecture decisions, design philosophy, and CLI documentation live under `docs/`.

The `normalize-syntax-rules` crate provides the syntax linting engine with 94 builtin rules covering Rust (13), JavaScript (10), TypeScript (5), Python (12), Go (9), C/C++ (4), Java (6), C# (6), Kotlin (5), Swift (5), PHP (5), Ruby (9), and cross-language (4). Severity is promoted to error for genuine bugs (mutable default args, bare except, `dbg!`/`todo!` macros, eval) and warning for style issues. Language query coverage lives in `normalize-languages/src/queries/`: approximately 335 `.scm` files covering 84 languages across 5 query types (tags, calls, complexity, imports, types). Every directory with source files has a `SUMMARY.md`, enforced at `severity=error` via `normalize rules run --engine native` in the pre-commit hook.

The `normalize-ratchet` crate tracks metric regressions against a baseline (line counts, function counts, complexity, call-graph complexity) using `normalize ratchet check/update/add/remove`. The `normalize-budget` crate enforces diff-based limits on code growth using `normalize budget check/add/update`. Both share metric primitives from `normalize-metrics`. The `normalize-native-rules` crate implements built-in rules including stale-summary, stale-docs, broken-ref, and check-examples checks, plus ratchet and budget integration for `normalize rules run --engine native`. The Datalog rules engine uses `ascent-interpreter` (v0.1.2) with incremental evaluation via `normalize-facts-rules-interpret`. The daemon (`daemon.rs`) primes the per-root `ENGINE_CACHE` after each index refresh by calling `normalize_rules::collect_fact_diagnostics_incremental` with the affected-file set; subsequent `normalize rules run` invocations in the same process use the incremental path (retract + re-derive only affected strata) instead of a cold full evaluation. All service methods use `display_with` for output routing; report structs use the `Report` suffix and signal errors via `Err(...)` rather than a `success: bool` field. Diagnostic output uses `tracing` throughout — `eprintln!` is not used in service code.
