# Normalize Monorepo

Normalize is a structural code intelligence CLI: index symbols and calls, enforce rules, track complexity — across any language. The monorepo contains 44 published Rust crates plus three `publish = false` crates (`normalize-grammars`, `xtask`, `benches`). The publish workflow (`.github/workflows/publish.yml`) handles crates.io's "too many new crates" 429 rate limit by parsing the Retry-After timestamp and sleeping until then before retrying — first-time publishes of large crate sets no longer fail mid-run. All SQLite access goes through `libsql` exclusively across the workspace (the rusqlite/libsql co-link conflict was resolved by migrating the three former rusqlite consumers — ca_cache, native-rules cache, syntax-rules runner — onto libsql with embedded `block_on` shims to keep their public APIs synchronous). Top-level CLI subcommands: `view`, `grep`, `context`, `structure`, `rank`, `trend`, `analyze`, `syntax`, `rules`, `budget`, `ratchet`, `ci`, `edit`, `kg`, `sessions`, `tools`, `serve`, `generate`, `package`, `docs`, `config`, `daemon`. `normalize docs` fetches upstream symbol documentation into LLM context across ecosystems (Rust/Go/Python): it dispatches over the `Ecosystem` trait, resolving docs from installed local source first (Cargo source, Go module cache, Python site-packages) and falling back to the upstream package source archive (docs.rs, Go module proxy `.zip`, PyPI sdist) for uninstalled packages, with `--ecosystem`/`-e` to select or disambiguate; doc bodies are stored source-native (`doc_body` + `doc_format`) and cached in the knowledge graph. The `kg` subcommand provides a persistent knowledge graph with three primitives: `read` (selector → units), `write` (jq transform → mutate/delete), `walk` (BFS graph traversal). `normalize trend` provides time-series health metrics over git history: `trend multi`, `trend complexity`, `trend length`, `trend density`, `trend test-ratio`., a VS Code extension (`editors/`), a web sessions viewer (`web/`), a documentation site (`docs/`), and supporting scripts and tooling. The `benches/` directory holds a Criterion benchmark suite covering `structure rebuild`, `rules run`, and CLI commands, with baseline numbers tracked in `docs/perf-baseline.md`. The main `normalize` binary consumes a library ecosystem of domain crates covering language support, facts extraction, rules evaluation, manifest parsing, and output formatting. Core development conventions are in `CLAUDE.md`; architecture decisions, design philosophy, and CLI documentation live under `docs/`. All 22 `normalize rank` subcommands now conform to the house style documented in `docs/cli-design.md` ("Rank output house style"): `#`-prefixed title with inline summary stats, `format_ranked_table` for tabular commands, `nu_ansi_term` for pretty mode, bare integers throughout, no footnotes in body text. The shared rank/render layer (the `Entity` trait, `RankEntry`, `RankStats`, `rank_pipeline`, `format_ranked_table`, `truncate_path`) lives in the `normalize-rank` crate (renamed from the misleadingly-named `normalize-analyze` in 2026-07 — it is rank/render infrastructure, not an analysis engine).

The `normalize-syntax-rules` crate provides the syntax linting engine with 95+ builtin rules covering Rust (14, including `rust/missing-module-doc`), JavaScript (10), TypeScript (5), Python (12), Go (9), C/C++ (4), Java (6), C# (6), Kotlin (5), Swift (5), PHP (5), Ruby (9), and cross-language (4). Severity is promoted to error for genuine bugs (mutable default args, bare except, `dbg!`/`todo!` macros, eval) and warning for style issues. Language query coverage lives in `normalize-languages/src/queries/`: approximately 335 `.scm` files covering 84 languages across 5 query types (tags, calls, complexity, imports, types). CFG queries (`.cfg.scm`) cover 76 languages with Phases 1–4 complete: Phase 1 (structural CFG, Mermaid renderer, `normalize cfg`), Phase 2 (def/use sites, liveness analysis, `normalize analyze liveness`), Phase 3 (effect captures `@cfg.effect.await/defer/yield/acquire/release/send/receive`, `normalize analyze effects`), Phase 4 (type-refined exception flow, `normalize analyze exceptions`). SQLite schema v15. The `normalize-refactor` crate provides `normalize edit extract-function` (plus add-parameter / inline-variable / introduce-variable) — liveness-based parameter/return inference with effect and exception boundary checks. Per-language source synthesis is delegated to the `RefactorCodeGen` trait in `normalize-languages` and structural node classification to `<lang>.refactor.scm` queries; supported languages are those that have both (canonical list in `normalize-languages/src/queries/SUMMARY.md` — currently rust, python, javascript, typescript, tsx, go, java, ruby, lua, kotlin, swift, dart, c#, php, scala, groovy, vb, zig, d — with lisp/ML/pattern-equation and logic/array languages deferred per TODO.md). Every directory with source files has a `SUMMARY.md`, enforced at `severity=error` via `normalize rules run` in the pre-commit hook.

Phase 0 cross-file name resolution is complete: `ModuleResolver` implementations for 26 languages (Rust, TypeScript/TSX, JavaScript, Python, Go, Ruby, Java, Kotlin, Groovy, Scala, C#, VB, F#, Swift, Dart, Zig, Elixir, Erlang, Haskell, OCaml, Lua, PHP, Perl, Clojure, Common Lisp, Scheme, Gleam, ReScript) are wired into `normalize structure rebuild` via `resolve_imports_via_module_resolver()`. `find_references` results are tagged with `confidence: "resolved" | "heuristic"`. New Datalog predicates: `resolved_import`, `module`, `export`, `reexport`, `symbol_use`, `resolved_reference`, `resolved_call`, `module_search_path`. Remaining blockers for 0.4.0 include resolvers for C/C++/ObjC, Elm, D, R, Julia, MATLAB, Nix, Ada, Agda, Idris, Lean, Prolog, and the full LSP surface (Phases 1–5). See `TODO.md`.

The `normalize-ratchet` crate tracks metric regressions against a baseline (line counts, function counts, complexity, call-graph complexity) using `normalize ratchet check/update/add/remove`. The `normalize-budget` crate enforces diff-based limits on code growth using `normalize budget check/add/update`. Both share metric primitives from `normalize-metrics`. The `normalize-native-rules` crate implements built-in rules: `missing-summary` (presence) and `stale-summary` (freshness) both support `filenames` and `paths` glob config; `stale-docs`, `broken-ref`, and `check-examples` provide additional checks; `long-file`, `high-complexity`, and `long-function` are advisory threshold rules backed by a SQLite findings cache (`.normalize/findings-cache.sqlite`) for warm-run performance; `high-fan-out` and `high-fan-in` are index-based coupling smell rules (default disabled, configurable threshold, tags: `architecture`, `coupling`) — `high-fan-out` flags files that import from too many distinct modules, `high-fan-in` flags files imported by too many distinct files. The Datalog rules engine uses `ascent-interpreter` (v0.1.5) with incremental evaluation via `normalize-facts-rules-interpret`; JIT is disabled pending an upstream arity-mismatch fix. The dylib rule pack system has been removed — it caused allocator boundary heap corruption; custom native rules now use an external-process model. The fact engine runs on a dedicated 64MB thread to avoid stack overflow from deep Datalog recursion.

The `normalize sessions` command surface (list/show/stats/analyze/messages/ngrams/patterns/heatmap/cost/plans/mark/…) lives in the standalone `normalize-sessions` crate, which owns the `#[cli] SessionsService`, report structs, and `OutputFormatter` impls and is mounted by the main service layer in one line — the "crate owns its subcommand" pattern (like `normalize-budget`/`normalize-cfg`); it depends on `normalize-chat-sessions` (parsing) and `normalize-session-analysis` (analysis). `normalize sync <dest>` copies a project dir + associated session metadata (via `project_metadata_roots` from `normalize-chat-sessions`) to a destination, with post-copy index path rewriting, `--dry-run`, `--all`, `--active N`, `--repo`/`--exclude` filters. The install script (`install.sh`) auto-selects the musl binary on NixOS. The release workflow (`.github/workflows/release.yml`) now builds tree-sitter grammars per-target — the musl matrix entry runs `cargo xtask build-grammars --target x86_64-unknown-linux-musl --cc musl-gcc` so the grammar `.so` files in the musl tarball are linked against musl libc and loadable by the bundled musl runtime. The musl artifact bundles `libgcc_s.so.1` sourced from Alpine Linux (musl-linked) rather than glibc's copy, fixing runtime failures on NixOS and non-FHS distros. `normalize update` now re-downloads grammars when the binary version differs from the installed stamp. The `flake.nix` devshell uses fenix for the Rust toolchain, merging the musl target `rust-std` in via `fenix.combine` (the `withComponents` helper only accepts host components); `nix build .#normalize-musl` validates the musl target locally. The toolchain is pinned at `1.95.0` via `rust-toolchain.toml` (components: rustc, cargo, clippy, rustfmt, rust-src; target: x86_64-unknown-linux-musl); the flake's `shellHook` diff-guards and auto-installs `.git/hooks/pre-commit` on every `nix develop` entry. The pre-commit hook hard-gates `cargo clippy --all-targets --all-features -- -D warnings` (was previously non-blocking).

The daemon (`daemon.rs`, Unix-only) maintains a `DiagnosticsCache` per watched root, caching syntax, fact, and native rule diagnostics. The cache is primed eagerly on file changes (syntax rules run incrementally on affected files, fact rules use the `ENGINE_CACHE`, native rules do a full re-run) and lazily on first `RunRules` request. `normalize rules run` routes through the daemon for cached results when available, falling back to local evaluation transparently; it returns a typed `DiagnosticsReport` supporting `--json`, `--jq`, `--jsonl`, `--sarif`, and `--pretty` output. The `normalize ci` command provides a single entry point for CI. The rules engine supports four engines: syntax (tree-sitter queries), native (Rust), fact (Datalog), and sarif (external tools emitting SARIF 2.1.0 via `[[rules.sarif-tools]]` config). All service methods use `display_with` for output routing; report structs use the `Report` suffix. Text-vs-pretty selection is per-command and resolved root-aware via `resolve_format(root)`: as of the server-less 0.6 `CliGlobals` migration (2026-06-29) each service implements one `CliGlobals` sink that stashes the `--pretty`/`--compact` globals into `pretty_raw`/`compact_raw` cells, replacing the per-method param wiring that previously let methods silently fall back to text (audit trail in `docs/artifacts/sessions-stats-output-2026-06-20/`). A consolidated CLI-surface audit (structured output, flag naming, dry-run coverage, exit codes, command structure) with a prioritized triage lives in `docs/artifacts/cli-audit-2026-06-29/`. Diagnostic output uses `tracing` throughout.

Command discoverability: all major service methods in `src/service/` now include "Also known as:" synonym paragraphs in their `///` doc strings, so users searching for "blast radius", "dead code", "find usages", "call hierarchy", "co-change", "orphan", "circular imports", etc. can locate the right command via `--help`. Command aliases in `main.rs` map familiar names from other tools: `search`/`find` to `grep`, `lint` to `rules run`, `check` to `ci`, `index` to `structure rebuild`, `refactor` to `edit`. The `normalize guide <topic>` system (topics: `setup`, `explore`, `analyze`, `rules`, `tree-sitter`) provides task-oriented walkthroughs whose example command references are kept in sync with the live CLI surface (README, LLMS.md, and guide text track the canonical command names — e.g. ranking commands live under `rank`, dependency/call-graph views under `view`).

The `normalize context` command implements a general-purpose contextual text resolution system: `.normalize/context/` directories are walked bottom-up (project-specific first, global `~/.normalize/context/` last), `.md` files are parsed into blocks with optional YAML frontmatter, and blocks are filtered against caller-provided context (`--match KEY=VALUE`, `--stdin --prefix NAMESPACE` JSON, `--all`). Conditions support composable `all:/any:` groups with pluggable match strategies: `equals`, `contains`, `keywords`, `regex`, `exists`, `one_of`. Dot-path keys address nested YAML frontmatter (e.g. `claudecode.hook`). The design spec is in `docs/context-redesign.md`. A deduplication fix ensures `~/.normalize/context/` is not visited twice when the working directory is inside the home directory.

The `normalize-git` crate was extracted from the main crate in 2026-07: verbatim `open_repo`, `read_blob_text`, `walk_tree_at_ref`, and related git utilities are now shared across budget, ratchet, semantic, and the main crate via `normalize-git`; per-crate `git_ops.rs` duplications were converted to thin re-export wrappers or removed. The graph-crate split decision is recorded in `docs/artifacts/cli-taxonomy-2026-06-29/DECISION-graph-crate.md`: the generic half (datatype-agnostic graph algorithms, zero normalize dependencies) is slated to become a standalone crate/repo; the normalize-flavored half (`GraphTarget`, `DependentEntry`, `BlastRadius`, `GraphReport`, all `OutputFormatter` impls) moves to dependency-analysis callers; `find_longest_chains`/`longest_path_from` duplication between normalize-graph and normalize-architecture is a known bug to consolidate. All mutating commands now have `--dry-run` coverage (audit T1-2). The vendored CLI drop-ins (`src/rg/`, `src/ast_grep/`, `src/jq/`) stay in the main crate — extracting them into their own crates is impossible under a publish trilemma (purity vs. `cargo install normalize` shipping the drop-ins vs. not publishing verbatim third-party CLI copies; a published crate cannot depend on a `publish = false` crate), so keeping them in `normalize` is forced; see `docs/audit-2026-07-02.md` and `docs/cli-dropin-integrations.md`. The `normalize-semantic` crate (vector embeddings over symbol/doc/commit chunks) is published standalone but no longer wired into the `normalize` binary as of 0.3.0. Symbol search is being redesigned around discrete tags for a future release; see `TODO.md` for the sketch.

Grammar load failures during `structure rebuild` are now loud: `SymbolParser::parse_file` returns `Option<Vec<FlatSymbol>>` (None = grammar .so unavailable), both extraction paths warn once per grammar via `warn_grammar_unavailable_once` and skip the file rather than indexing it as empty. The CA cache and SQLite are never written for grammar-unavailable files.

The daemon IPC layer now uses rkyv binary serialization (magic-byte protocol, BLOB storage) replacing JSON for a significant latency reduction. The daemon maintains a persistent SQLite index connection per watched root, eliminating lock races under concurrent access. SARIF tool results use mtime-based caching via `watch` glob patterns. The findings cache uses WAL mode + single-transaction writes (~12× faster cold runs). Fact rule evaluation runs in parallel (~2× speedup). The daemon now implements LSP-style diagnostic push: per-file diagnostic storage and binary subscribe-channel deltas eliminate the poll-on-request model. `#match?` and `#eq?` tree-sitter predicates are now evaluated for language queries. `CONTEXT.md` documents the ubiquitous language used across the codebase. Post-edit index invalidation: refactoring commands (`rename`, `introduce-variable`, `inline-variable`, `add-parameter`, `move`, `inline-function`) push changed file paths to the daemon via `Request::FilesChanged` immediately after writing edits, triggering incremental re-indexing without waiting for inotify. Non-fatal if the daemon is not running.

Configuration semantics: `WalkConfig::default()` is empty per Rust convention; opinions live in `NormalizeConfig::bootstrap()` (currently `[walk] exclude = [".git/"]`), used by `NormalizeConfig::load` when no project config exists. `normalize init` starts from bootstrap and adds detected scratch directories (currently just `.claude/worktrees/`) to `[walk] exclude`, prompting per entry in interactive mode and adding them all in non-interactive mode. All walkers (native, syntax, rules) compile `[walk] exclude` patterns through `WalkConfig::compiled_excludes` and match relative paths via `Gitignore::matched_path_or_any_parents`, so patterns with slashes (e.g. `.claude/worktrees/`) match correctly.

The `tooling/claude-hooks/post-history.sh` PHI hook is now role-aware: it emits a shared core (5 confidence/momentum/framing/gap/decision principles), then branches on whether the current session is a subagent (inlined `is_subagent` top-level-agent-id detector) to emit either a subagent tail (gather-with-tools, calibrate-return) or a main-session tail (delegate-as-peer, trace-subagent-findings). The ecosystem-wide `tooling/propagate-post-history.sh` propagates the canonical from `rhizone/github-io`.
