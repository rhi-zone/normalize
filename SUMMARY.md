# Normalize Monorepo

Normalize is a structural code intelligence CLI: index symbols and calls, enforce rules, track complexity — across any language. The monorepo contains 38 published Rust crates plus three `publish = false` crates (`normalize-grammars`, `xtask`, `normalize-bench`), a VS Code extension (`editors/`), a web sessions viewer (`web/`), a documentation site (`docs/`), and supporting scripts and tooling. The `benches/` directory holds a Criterion benchmark suite covering `structure rebuild`, `rules run`, and CLI commands, with baseline numbers tracked in `docs/perf-baseline.md`. The main `normalize` binary consumes a library ecosystem of domain crates covering language support, facts extraction, rules evaluation, manifest parsing, and output formatting. Core development conventions are in `CLAUDE.md`; architecture decisions, design philosophy, and CLI documentation live under `docs/`.

The `normalize-syntax-rules` crate provides the syntax linting engine with 95+ builtin rules covering Rust (14, including `rust/missing-module-doc`), JavaScript (10), TypeScript (5), Python (12), Go (9), C/C++ (4), Java (6), C# (6), Kotlin (5), Swift (5), PHP (5), Ruby (9), and cross-language (4). Severity is promoted to error for genuine bugs (mutable default args, bare except, `dbg!`/`todo!` macros, eval) and warning for style issues. Language query coverage lives in `normalize-languages/src/queries/`: approximately 335 `.scm` files covering 84 languages across 5 query types (tags, calls, complexity, imports, types). Every directory with source files has a `SUMMARY.md`, enforced at `severity=error` via `normalize rules run` in the pre-commit hook.

The `normalize-ratchet` crate tracks metric regressions against a baseline (line counts, function counts, complexity, call-graph complexity) using `normalize ratchet check/update/add/remove`. The `normalize-budget` crate enforces diff-based limits on code growth using `normalize budget check/add/update`. Both share metric primitives from `normalize-metrics`. The `normalize-native-rules` crate implements built-in rules: `missing-summary` (presence) and `stale-summary` (freshness) both support `filenames` and `paths` glob config; `stale-docs`, `broken-ref`, and `check-examples` provide additional checks. The Datalog rules engine uses `ascent-interpreter` (v0.1.5) with incremental evaluation via `normalize-facts-rules-interpret`; JIT is disabled pending an upstream arity-mismatch fix. The dylib rule pack system has been removed — it caused allocator boundary heap corruption; custom native rules now use an external-process model. The fact engine runs on a dedicated 64MB thread to avoid stack overflow from deep Datalog recursion.

The daemon (`daemon.rs`, Unix-only) primes the per-root `ENGINE_CACHE` after each index refresh; subsequent `normalize rules run` invocations use the incremental path. The `normalize ci` command provides a single entry point for CI. The rules engine supports four engines: syntax (tree-sitter queries), native (Rust), fact (Datalog), and sarif (external tools emitting SARIF 2.1.0 via `[[rules.sarif-tools]]` config). All service methods use `display_with` for output routing; report structs use the `Report` suffix. Diagnostic output uses `tracing` throughout.
