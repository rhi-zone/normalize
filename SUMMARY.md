# Normalize Monorepo

Normalize is a structural code intelligence CLI: index symbols and calls, enforce rules, track complexity — across any language. The monorepo contains 38 published Rust crates plus three `publish = false` crates (`normalize-grammars`, `xtask`, `normalize-bench`). Top-level CLI subcommands: `view`, `grep`, `context`, `structure`, `rank`, `trend`, `analyze`, `syntax`, `rules`, `budget`, `ratchet`, `ci`, `edit`, `sessions`, `tools`, `serve`, `generate`, `package`, `config`, `daemon`. `normalize trend` provides time-series health metrics over git history: `trend multi`, `trend complexity`, `trend length`, `trend density`, `trend test-ratio`., a VS Code extension (`editors/`), a web sessions viewer (`web/`), a documentation site (`docs/`), and supporting scripts and tooling. The `benches/` directory holds a Criterion benchmark suite covering `structure rebuild`, `rules run`, and CLI commands, with baseline numbers tracked in `docs/perf-baseline.md`. The main `normalize` binary consumes a library ecosystem of domain crates covering language support, facts extraction, rules evaluation, manifest parsing, and output formatting. Core development conventions are in `CLAUDE.md`; architecture decisions, design philosophy, and CLI documentation live under `docs/`.

The `normalize-syntax-rules` crate provides the syntax linting engine with 95+ builtin rules covering Rust (14, including `rust/missing-module-doc`), JavaScript (10), TypeScript (5), Python (12), Go (9), C/C++ (4), Java (6), C# (6), Kotlin (5), Swift (5), PHP (5), Ruby (9), and cross-language (4). Severity is promoted to error for genuine bugs (mutable default args, bare except, `dbg!`/`todo!` macros, eval) and warning for style issues. Language query coverage lives in `normalize-languages/src/queries/`: approximately 335 `.scm` files covering 84 languages across 5 query types (tags, calls, complexity, imports, types). Every directory with source files has a `SUMMARY.md`, enforced at `severity=error` via `normalize rules run` in the pre-commit hook.

The `normalize-ratchet` crate tracks metric regressions against a baseline (line counts, function counts, complexity, call-graph complexity) using `normalize ratchet check/update/add/remove`. The `normalize-budget` crate enforces diff-based limits on code growth using `normalize budget check/add/update`. Both share metric primitives from `normalize-metrics`. The `normalize-native-rules` crate implements built-in rules: `missing-summary` (presence) and `stale-summary` (freshness) both support `filenames` and `paths` glob config; `stale-docs`, `broken-ref`, and `check-examples` provide additional checks; `long-file`, `high-complexity`, and `long-function` are advisory threshold rules backed by a SQLite findings cache (`.normalize/findings-cache.sqlite`) for warm-run performance. The Datalog rules engine uses `ascent-interpreter` (v0.1.5) with incremental evaluation via `normalize-facts-rules-interpret`; JIT is disabled pending an upstream arity-mismatch fix. The dylib rule pack system has been removed — it caused allocator boundary heap corruption; custom native rules now use an external-process model. The fact engine runs on a dedicated 64MB thread to avoid stack overflow from deep Datalog recursion.

`normalize sync <dest>` copies a project dir + associated session metadata (via `project_metadata_roots` from `normalize-chat-sessions`) to a destination, with post-copy index path rewriting, `--dry-run`, `--all`, `--active N`, `--repo`/`--exclude` filters. The install script (`install.sh`) auto-selects the musl binary on NixOS.

The daemon (`daemon.rs`, Unix-only) maintains a `DiagnosticsCache` per watched root, caching syntax, fact, and native rule diagnostics. The cache is primed eagerly on file changes (syntax rules run incrementally on affected files, fact rules use the `ENGINE_CACHE`, native rules do a full re-run) and lazily on first `RunRules` request. `normalize rules run` routes through the daemon for cached results when available, falling back to local evaluation transparently; it returns a typed `DiagnosticsReport` supporting `--json`, `--jq`, `--jsonl`, `--sarif`, and `--pretty` output. The `normalize ci` command provides a single entry point for CI. The rules engine supports four engines: syntax (tree-sitter queries), native (Rust), fact (Datalog), and sarif (external tools emitting SARIF 2.1.0 via `[[rules.sarif-tools]]` config). All service methods use `display_with` for output routing; report structs use the `Report` suffix. Diagnostic output uses `tracing` throughout.

Command discoverability: all major service methods in `src/service/` now include "Also known as:" synonym paragraphs in their `///` doc strings, so users searching for "blast radius", "dead code", "find usages", "call hierarchy", "co-change", "orphan", "circular imports", etc. can locate the right command via `--help`. Command aliases in `main.rs` map familiar names from other tools: `search`/`find` to `grep`, `lint` to `rules run`, `check` to `ci`, `index` to `structure rebuild`, `refactor` to `edit`.

The `normalize context` command implements a general-purpose contextual text resolution system: `.normalize/context/` directories are walked bottom-up (project-specific first, global `~/.normalize/context/` last), `.md` files are parsed into blocks with optional YAML frontmatter, and blocks are filtered against caller-provided context (`--match KEY=VALUE`, `--stdin --prefix NAMESPACE` JSON, `--all`). Conditions support composable `all:/any:` groups with pluggable match strategies: `equals`, `contains`, `keywords`, `regex`, `exists`, `one_of`. Dot-path keys address nested YAML frontmatter (e.g. `claudecode.hook`). The design spec is in `docs/context-redesign.md`. A deduplication fix ensures `~/.normalize/context/` is not visited twice when the working directory is inside the home directory.

The `normalize-semantic` crate adds a semantic retrieval layer: vector embeddings over structurally-derived chunks (symbol name + signature + doc comment + callers/callees + co-change neighbors), stored in SQLite alongside the structural index, queryable via `normalize structure search <query>`. Uses fastembed (ONNX-backed, no server) with `nomic-embed-text-v1.5` as default model. Disabled by default; enable with `embeddings.enabled = true` in `.normalize/config.toml`. Embeddings are generated at the end of `structure rebuild` and re-ranked at query time using cosine similarity + staleness penalty.
