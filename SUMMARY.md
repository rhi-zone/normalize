# Normalize Monorepo

Normalize is a polyglot code intelligence CLI toolchain providing structural analysis of codebases through AST-based extraction. The monorepo contains 38 Rust crates (published on crates.io), a VS Code extension, a web sessions viewer, documentation site, and supporting tooling. The main `normalize` binary consumes a library ecosystem of domain crates covering language support, facts extraction, rules evaluation, manifest parsing, and output formatting. Core development conventions are in `CLAUDE.md`; architecture decisions, design philosophy, and CLI documentation live under `docs/`.

The `normalize-syntax-rules` crate provides the syntax linting engine; 94 builtin syntax rules cover Rust (13), JS (10), TS (5), Python (12), Go (9), C/C++ (4), Java (6), C# (6), Kotlin (5), Swift (5), PHP (5), Ruby (9), and cross-language (4), with severity promotion — error for genuine bugs (mutable default args, bare except, dbg!/todo! macros, eval), warning for style — and fix-fixture infrastructure for testing auto-fix transforms. `run_rules()` takes separate `project_root` and `root` (scan target) parameters so allow-list path matching works correctly when scanning a subdirectory.

Every directory with source files has a `SUMMARY.md` — enforced at `severity=error` via `normalize rules run --engine native` in the pre-commit hook (native engine covers stale-summary, check-refs, stale-docs, check-examples). The `service/` layer (server-less `#[cli]`) is the primary CLI registration point; all `cmd_*` i32-returning wrappers have been eliminated from both `commands/` and `normalize-rules/src/` — service methods return `Result<T, String>` with natural `?` propagation. `normalize analyze check` deleted — subsumed by `normalize rules run --engine native`. `normalize facts` renamed to `normalize structure` (CLI command name now matches `#[cli(name = "structure")]`; server-less nested service `name` attribute was being ignored in favor of field name — workaround: rename field). Rule orchestration (`RulesService`, `run_rules_report()`, `apply_native_rules_config()`, syntax rule runner) has been extracted into the `normalize-rules` crate. `normalize rules list` returns a structured `RulesListReport` (Serialize+JsonSchema) covering all three engines (syntax/fact/native), with `OutputFormatter` pretty/compact rendering. `main()` is `#[tokio::main] async fn`; service methods are `async fn`; sync-trait bridge code (`path_resolve.rs`, `extract.rs`, `health.rs`) uses `block_in_place`+`Handle::try_current` instead of `Runtime::new()`. Tree-sitter introspection: `normalize analyze parse/query/node-types` for CST viewing, query testing, and grammar exploration. `normalize guide <topic>` provides workflow-oriented guides (rules, explore, setup, analyze, tree-sitter).

Three sub-crates (`normalize-facts`, `normalize-filter`, `normalize-syntax-rules`) now ship standalone CLI binaries behind a `cli` feature, with `dep:clap` dropped (server-less re-exports clap). Service framework upgraded to server-less 0.4.0; `about =` renamed to `description =` in all `#[cli]` attributes. The main `normalize` crate uses `cli-full = ["cli", "jq-cli", "rg-cli", "ast-grep-cli"]` as default, with drop-in wrappers each implying `cli`.

Language query coverage in `normalize-languages/src/queries/`: ~335 `.scm` files covering 84 languages across 5 query types (tags, calls, complexity, imports, types). Goal: maximum quality for every supported language. Grammars come from arborium or we write our own (Jinja2 precedent). `grammars/jinja2/` — high-quality Jinja2 grammar written from scratch (54 corpus tests, full expression language, distinct named node types per statement kind); installed locally, pending publish as our own crate. All query files are registered in `grammar_loader.rs`. 47/84 languages at 100% .scm coverage; ongoing work to close remaining gaps. `docstring.rs` adds `extract_preceding_prefix_comments` shared by Lua and R. `body.rs` `analyze_brace_body`/`analyze_paren_body` delegate to a shared `analyze_delimited_body`. `arch_common.rs` in `normalize-package-index` adds `parse_arch_db`/`parse_arch_db_desc` shared by CachyOS and EndeavourOS. `DepsExtractor` uses a query-first import path (`collect_imports_from_query` via `*.imports.scm`) with trait-based fallback. `type_refs` table stores deep type edges for Rust, TypeScript/TSX, Python, Go, Java, C#, Kotlin, Swift, C++, Ruby. Extraction bugs fixed for Elixir (no-args defs), Haskell (dedup), Kotlin (property_declaration short-circuit). Multi-repo support (`--repos-dir`) on all analyze and tools commands — standardized flag name across ownership, hotspots, duplicates, contributors, activity, repo_coupling, cross_repo_health, tools lint/test. Rule configuration migrated from `[analyze.rules]` to `[rules]`; `normalize-rules-config` holds canonical types. 257 fixture tests across 68 languages.

`normalize-filter` detects bare language names passed to `--only`/`--exclude` (e.g. `--only rust`) and emits a helpful error instead of silently matching nothing. `normalize analyze complexity` and `normalize analyze length` now support single-file input, routing to `analyze_file_*` directly instead of requiring a directory. `normalize analyze complexity-trend`, `length-trend`, `density-trend`, `test-ratio-trend` — scalar time-series commands using git worktrees. `normalize analyze complexity` and `length` support `--diff <ref>` for annotated delta views against a baseline commit. Generic `--diff` infrastructure via `DiffableRankEntry` trait in `normalize-analyze::ranked` enables any rank command to add baseline comparison; `test-ratio`, `density`, and `uniqueness` use it. Import extraction fixes: Groovy `groovy.imports.scm` updated to use `import: (qualified_name)` field accessor (was using invalid `dotted_identifier` node kind); `DepsExtractor::collect_imports_from_query` now skips empty-path sentinel matches and returns `None` when empty (fixing Scala ghost imports and ensuring correct trait fallback for both languages).

`normalize view` refactored (phase 2): `ViewOutput` enum and all per-kind report structs (`ViewFileReport`, `ViewSymbolReport`, `ViewLineRangeReport`, etc.) dissolved into a single unified `ViewReport` struct containing a `ViewNode` tree; `normalize view list` added as a new subcommand returning `ViewListReport(Vec<ViewReport>)` for multi-result views. `normalize view` now surfaces warnings when a language doesn't support the requested capability — data/config languages (JSON, YAML, TOML, HTML, CSS, XML) show "symbol extraction is not supported" rather than silently returning empty symbols. Warnings appear in text, pretty, and JSON output via the `ViewReport.warnings` field.

`normalize config schema/show/validate/set` — generic schema-driven config UI. `show` walks the JSON Schema to display all available options with descriptions and types; `--section` accepts dotted paths; `--set-only` hides unset fields. `set` validates against the schema before writing and blocks on errors; `--force` bypasses; `--dry-run` previews. Uses `jsonschema` crate for validation, `toml_edit` for typed writes. `normalize daemon watch [path]` — new command that streams file-change and index-refresh events from the daemon in `[HH:MM:SS] modified <file>` / `[HH:MM:SS] index refreshed (N files)` format until Ctrl-C. Backed by `tokio::sync::broadcast` event channel in `daemon.rs`; `DaemonClient::watch_events()` for blocking subscription.

`normalize analyze similar-functions` now suppresses widespread body-pattern clusters: connected-component analysis on the pair graph identifies clusters spanning 3+ distinct files with 5+ pairs (different method names, same body) and folds them into `SuppressedBodyPatternGroup` summary entries instead of surfacing O(N²) noise. Catches the Language trait impl pattern (e.g. `extract_imports` identically implemented across 20 language structs). Applied when `!include_trait_impls`. Updated 2026-03-15.

`normalize sessions messages` improvements (2026-03-17): output now grouped by session (single `[id] project  date` header per session; messages indented with `[role] HH:MM:SS  text`); `--role` accepts `tool`, `system`, `all` in addition to `user`/`assistant`; `--grep` searches through tool call parameters (`ToolUse` name+input JSON) and thinking blocks in addition to text/tool-result content; invalid `--role` values show a descriptive error instead of the generic clap message. Depends on server-less 0.4.7 (fixes `SchemaValueParser` to surface `FromStr` errors via `Error::raw(ValueValidation)`; intra-workspace version constraints now use workspace dependency inheritance). jaq-json bumped to 2.0.0-gamma: `Colors` → `Styles`, `Val::Str(s, _)` → `Val::BStr(s) | Val::TStr(s)`.

`normalize sessions patterns` (2026-03-20): Markov chain transition matrix analysis of tool call sequences across sessions. Computes population-level and per-session transition matrices (probability of tool A → tool B), detects outlier sessions via Frobenius norm divergence from the population matrix, and shows most common starting/ending tools. Supports same filters as other session commands (`--mode`, `--agent-type`, `--days`, `--grep`, etc.).

`normalize-metrics` (2026-03-22): shared metric primitives extracted from `normalize-ratchet` for reuse by `normalize-ratchet` and `normalize-budget`. Contains `Metric` trait, `MetricFactory` type alias, `Aggregate` enum with `aggregate()` function, and `filter_by_prefix()` path filtering utility.

`normalize-ratchet` (in progress, 2026-03-22): metric regression-tracking system. Stores a baseline of aggregated metric values in `.normalize/ratchet.json`; flags when current values regress past the baseline. Entries are `(path, metric, aggregation) → value` where path can be a directory, file, or symbol address (`file/Parent/fn`) — aggregation is always applied (mean/median/max/min/sum/count). Metrics: cyclomatic complexity, call-graph complexity, line count, function count, class count, comment line count. Commands: `measure` (inspect), `add` (pin baseline), `check` (detect regressions; also runs as a native rule), `update` (re-pin), `show`. Now depends on `normalize-metrics` for shared primitives.

`normalize-budget` (2026-03-22): diff-based budget system. Checks whether changes (relative to a git ref) exceed configured limits. Each budget entry tracks how much a path is allowed to change: `max_added`, `max_removed`, `max_total`, `max_net`. Budget file at `.normalize/budget.json`. Diff metrics: lines, functions, classes, modules, todos, complexity-delta, dependencies. Commands: `measure`, `add`, `check`, `update`, `show`, `remove`. Runs as a native rule alongside ratchet.
